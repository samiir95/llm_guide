Section 01
How LLMs Work
Tokens, tokenization, transformer mechanics, and context windows.
What is an LLM?

A Large Language Model is a neural network trained to predict the next token in a sequence. It's not "thinking" — it's pattern-matching at massive scale. Given a prompt, it samples the most likely continuation.

"The sky is"Input prompt
→
Tokenizertext → IDs
→
Transformerattention layers
→
Logitsprobability dist
→
"blue"Sampled token
Tokens

Tokens are the atomic units of text an LLM sees. They are not words — they're subword fragments produced by a tokenizer (usually BPE or WordPiece).

# tiktoken (GPT-4 tokenizer)
"Hello, world!"  → [9906, 11, 1917, 0]   # 4 tokens
"unbelievable"   → [8416, 39110]          # 2 tokens
"GPT"            → [38, 2898, 51]           # 3 tokens

Rule of thumb: ~1 token ≈ 0.75 words in English. Code is more token-dense.

Tokenization

The tokenizer converts raw text → token IDs (integers) before the model sees anything. Each model has its own vocabulary (e.g. GPT-4: 100k vocab, Llama 3: 128k vocab).

Why it matters for you: API pricing is per-token. A 1,000-word test description ≈ ~1,300 tokens. Long JSON payloads are expensive.

⚡ Tokenization quirk
Numbers like 12345 might split into multiple tokens. "1", "23", "45" — causing math errors in small models.
Context Window

The context window is the total number of tokens a model can "see" at once — both input and output combined. It's the model's working memory.

ModelContext WindowNotes
GPT-4o128k tokens~96k words
Claude Sonnet 4.6200k tokens~150k words
Llama 3.1 8B128k tokenslocal / open-weight
Gemini 1.5 Pro1M tokenscan fit entire codebases
⚠ The "Lost in the Middle" problem
Models struggle to recall information placed in the middle of very long contexts. Critical content should be at the start or end of the prompt.
Attention & Transformers (simplified)

At the core of every LLM is the self-attention mechanism: every token can attend to (look at) every other token in the context. This is how "The bank by the river" vs "The bank approved the loan" disambiguates.

The model is a stack of these attention layers + feed-forward layers. More layers + more attention heads = more capacity to learn complex patterns during training.

Section 02
Model Types
Base vs instruct, reasoning vs non-reasoning, multimodal.
Base Model

A raw pretrained model that learned to predict the next token from massive internet text. It completes text — it does not follow instructions.

# Base model behavior
Prompt:  "The capital of France is"
Output:  "Paris. The capital of Germany is Berlin. The capital of..."
# Just keeps going — no stop, no intent

Used as a foundation for fine-tuning. Rarely used directly in apps. Example: Llama 3 base, Mistral base.

Instruct / Chat Model

A base model fine-tuned with RLHF (or DPO/PPO) on instruction-following data. It understands roles (system/user/assistant) and responds helpfully.

# Instruct model behavior
System:  "You are a helpful assistant."
User:    "What's the capital of France?"
Output:  "The capital of France is Paris."

This is what you use in production. Example: Llama-3-8B-Instruct, Claude Sonnet, GPT-4o.

Reasoning Model

Models trained to produce a chain-of-thought scratchpad before answering. They "think out loud" internally, leading to much better performance on math, logic, and multi-step problems.

Examples: o3, Claude 3.7 Sonnet (extended thinking), DeepSeek-R1, QwQ.

🧠 Trade-off
Reasoning models are slower and more expensive. Don't use them for simple classification or summarization tasks.
Non-Reasoning Model

Standard instruct models that respond directly. Fast, cheap, excellent for most tasks: summarization, classification, code generation, chat.

Examples: GPT-4o mini, Claude Haiku, Llama 3.1 8B Instruct.

Rule of thumb: start with a non-reasoning model. Only escalate to reasoning when you hit accuracy limits on complex logic.

Multimodal Models

Models that can process multiple input modalities beyond text — images, audio, video, and/or documents.

ModelModalitiesUse case
GPT-4oText + Image + AudioVision Q&A, voice
Claude Sonnet 4.6Text + Image + PDFDoc analysis, screenshot testing
Gemini 1.5 ProText + Image + Audio + VideoVideo understanding
LLaVA / Qwen-VLText + ImageLocal open-weight vision
💡 QA use case
Multimodal models can analyze screenshots of your app under test — useful for visual regression checks or screenshot-diff explanations.
Section 03
Parameters & VRAM
Size vs. performance vs. hardware requirements.
What are Parameters?

Parameters are the learned numerical weights of the model — billions of floating-point numbers. More parameters = more capacity to memorize patterns from training data.

# Example: Llama 3.1 8B
8,030,261,248 parameters (~8 billion)
Each stored as float16 (2 bytes) = ~16 GB VRAM to run at full precision
VRAM Requirements
ModelParamsfp16 VRAM4-bit Quant VRAMNotes
Llama 3.2 1B1B~2 GB~0.8 GBRuns on CPU!
Llama 3.1 8B8B~16 GB~5 GBRTX 3080 (10GB) ✗, RTX 4090 ✓

📐 Quick formula
VRAM (GB) ≈ (params × bytes_per_param) / 1e9 + ~20% overhead

fp32 = 4B/param, fp16/bf16 = 2B/param, int8 = 1B/param, int4 = 0.5B/param
Bigger ≠ Always Better

Scaling laws are real — but there are sharp diminishing returns beyond a task's "difficulty ceiling".

TaskGood enough modelWhy go bigger?
JSON extraction from text7B instructRarely needed
Summarization7–13BVery long docs only
Code generation (simple)7–13BUnless complex refactors
Multi-hop reasoning70B+Smaller models hallucinate chains
Medical / legal accuracy70B+ or fine-tunedHigh stakes domain
⚠ The Goldilocks principle
A 7B model fine-tuned on your domain can outperform a 70B generic model. Match model to task complexity + fine-tune if needed.
Section 04
Quantization & Formats
GGUF, safetensors, Ollama, vLLM — what, why, and when.
Quantization

Quantization reduces the precision of model weights to save memory and speed up inference, at the cost of minor quality degradation.

PrecisionBits/paramQualityUse case
float3232ReferenceTraining
bfloat1616≈fp32GPU inference
int8 (Q8)8Very goodServer inference
int4 (Q4)4GoodConsumer GPU/CPU
int2 (Q2)2DegradedExtreme memory constraints
# Llama 3.1 8B memory comparison
bf16:  16 GB   # full precision
Q8_0:  8.5 GB  # barely noticeable difference
Q4_K_M:4.9 GB  # best quality/size ratio — recommended
Q2_K:  3.2 GB  # noticeable quality loss
GGUF Format

GGUF (successor to GGML) is a single-file format for running quantized models on CPU/GPU via llama.cpp. It's what Ollama uses under the hood.

# GGUF filename encodes the quant level
llama-3.1-8b-instruct.Q4_K_M.gguf
#                         ^^^^^^ quantization type

Best for: local desktop inference, Ollama, no GPU needed (CPU offload).

Safetensors Format

Safetensors is Hugging Face's safe, fast-loading format for full-precision weights. Used with transformers, vLLM, and fine-tuning pipelines.

# Typical HuggingFace model repo
model-00001-of-00004.safetensors
model-00002-of-00004.safetensors
config.json
tokenizer.json

Best for: server GPU inference, fine-tuning, vLLM.

Ollama

Ollama wraps llama.cpp into a Docker-friendly CLI + local REST API. Dead simple for local development.

ollama pull llama3.1:8b
ollama run llama3.1:8b
# OpenAI-compatible API on localhost:11434
curl localhost:11434/api/chat \
  -d '{"model":"llama3.1","messages":[...]}'

Great for local dev, CI experiments, AI-assisted IDE plugins (Continue.dev).

vLLM

vLLM is a high-throughput production inference server with PagedAttention for efficient KV-cache management. Handles many concurrent requests efficiently.

vllm serve meta-llama/Llama-3.1-8B-Instruct \
  --tensor-parallel-size 2
# OpenAI-compatible: /v1/chat/completions

Use in production on GPU servers. Supports continuous batching, LoRA adapters, multi-GPU.

💡 Decision guide
Local dev/experiments → Ollama | Production self-hosted GPU → vLLM | Cloud API → OpenAI/Anthropic/Together
Section 05
Prompting Techniques
ChatML, roles, few-shot, chain-of-thought, parameters, structured output.
ChatML — The Message Format

ChatML is the structured format most instruct models use internally to separate roles. APIs abstract this — you pass JSON, the SDK applies the format.

# What you write (API)
{
  "messages": [
    {"role": "system",    "content": "You are a QA engineer."},
    {"role": "user",      "content": "Write a test for login."},
    {"role": "assistant", "content": "@Test public void..."}
  ]
}

# What the tokenizer sees (ChatML tokens)
<|im_start|>system
You are a QA engineer.<|im_end|>
<|im_start|>user
Write a test for login.<|im_end|>
Zero-shot Prompting

No examples — just instructions. Works well for capable models on clear tasks.

System: "Classify sentiment."
User:   "The deployment pipeline broke again."
Output: "Negative"
Few-shot Prompting

Provide 2–5 input/output examples in the prompt to steer the output format and behavior.

User: """
Q: "Login failed" → Negative
Q: "All tests green!" → Positive
Q: "Pipeline is slow" → """
Output: "Negative"
Chain-of-Thought (CoT)

Ask the model to reason step-by-step before answering. Dramatically improves accuracy on multi-step problems.

# Without CoT — often wrong on complex logic
User: "If test suite A takes 12min and B takes 8min and they share 3 workers, total time?"

# With CoT
User: "Think step by step. If test suite A takes 12min..."
Output: "Step 1: A needs 12min, B needs 8min.
Step 2: 3 workers can run both in parallel...
Step 3: Total wall time = max(12,8) = 12min + handoff ≈ 13min"
💡 Tip
Append "Let's think step by step." or "Think carefully before answering." for instant CoT activation without examples.
Generation Parameters
temperature 0.0 → 2.0
deterministiccreative / chaotic
0.0 = always picks the most likely token (good for JSON, code). 0.7–1.0 = balanced creative responses. >1.5 = random/experimental.
top_p (nucleus sampling) 0.0 → 1.0
narrowfull vocab
Only sample from tokens whose cumulative probability ≥ top_p. 0.9 = standard. Reduce for factual tasks. Don't set both temperature and top_p aggressively at the same time.
max_tokens
Hard cap on output length. The model stops generating once it hits this. Set it based on your expected output: 256 for short answers, 2048+ for long code, 8192 for full files.
stop sequences
Force the model to stop generating when it produces a specific string. E.g., ["```", "---", "\n\n"]. Very useful for constraining code blocks or structured outputs.
Structured Output / JSON Mode

Force the model to output valid JSON. Most APIs support this natively — it's applied at the token-sampling level.

# OpenAI / compatible APIs
{
  "response_format": {"type": "json_object"},
  "messages": [{"role": "user", "content":
    "Extract: name, severity, status from this bug report: ..."}]
}

# Output (guaranteed valid JSON)
{
  "name": "NullPointerException in PaymentFlow",
  "severity": "critical",
  "status": "open"
}
💡 QA automation use case
Use JSON mode to parse test failure logs into structured objects → feed into your ReportPortal or Jira integration.
Section 06
Context Engineering
What you put in the context window is the most impactful engineering decision you make.
What is Context Engineering?

Context engineering is the practice of deliberately constructing the LLM's input to maximize the quality of its output. It's prompt engineering + data retrieval + conversation state management combined.

System Promptpersona, rules
+
Retrieved DocsRAG / tools
+
Conv. Historytrimmed
+
User Messagecurrent turn
=
Context Windowwhat model sees
Key Principles
  • Relevance over size — cut noise ruthlessly
  • Recency bias — models favour recent context
  • Position matters — critical info at top or end
  • Format helps — headers/XML tags improve parsing
  • Summarize history — don't let conversations grow unbounded
Practical Techniques
  • Use <document>, <context> XML tags to delineate sections
  • Trim old conversation turns, keep only last N relevant turns
  • Inject only relevant retrieved docs, not all of them
  • For long codebases: send diffs, not full files
  • Re-state the task after injecting long context
System Prompt Architecture
<system>
You are a senior QA automation engineer at D360.
Your stack: Java, TestNG, RestAssured, ReportPortal.

<rules>
- Always output valid Java code
- Use @BeforeMethod / @AfterMethod appropriately
- Never skip null checks on API responses
</rules>

<context>
{injected_relevant_test_file_snippet}
</context>
</system>

Tool Use & MCP
Function calling, structured tool definitions, and the Model Context Protocol.
Function Calling

You define tools (functions) with a JSON schema. The model decides when to call them and returns a structured call — your code executes it and feeds back the result.

User message"What's the test pass rate?"
→
Model decidescall get_metrics()
→
Your code runsAPI/DB call
→
Result injectedback to model
→
Final answer"Pass rate: 94.2%"
# Tool definition (OpenAI format)
{
  "type": "function",
  "function": {
    "name": "get_test_metrics",
    "description": "Fetch pass/fail metrics for a test suite",
    "parameters": {
      "type": "object",
      "properties": {
        "suite_name": {"type": "string"},
        "date_range": {"type": "string"}
      },
      "required": ["suite_name"]
    }
  }
}

Section 08
Embeddings & RAG
Semantic search, vector databases, retrieval-augmented generation.
Embeddings

An embedding is a numerical vector representation of text. Semantically similar text produces vectors that are close together in high-dimensional space. This enables semantic search — finding relevant content by meaning, not keyword match.

# text-embedding-3-small (OpenAI)
"test failed due to timeout"     → [0.023, -0.187, 0.445, ...]  # 1536 dims
"assertion error in login flow"  → [0.031, -0.201, 0.412, ...]  # similar!
"quarterly revenue report Q3"    → [-0.412, 0.083, -0.219, ...] # very different

cosine_similarity(vec1, vec2) → 0.94  # high = semantically related
RAG — Retrieval-Augmented Generation

RAG combines a vector search with an LLM. Instead of relying on the model's baked-in knowledge (which has a cutoff), you retrieve relevant documents at query time and inject them into the context.

User query
→
Embed query
→
Vector searchtop-k similar docs
→
Inject into promptas context
→
LLM answersgrounded in docs
# Minimal RAG pipeline (pseudo-code)
query = "Why did the BrowserStack tests fail yesterday?"
query_vec = embed(query)                        # embed the question
docs = vector_db.search(query_vec, top_k=5)   # find similar test logs

prompt = f"""
<context>
{chr(10).join(docs)}
</context>
Answer based on the above context: {query}
"""
response = llm.complete(prompt)

Section 09
Agents & Orchestration
Agentic loops, multi-agent systems, and orchestration frameworks.
What is an Agent?

An agent is an LLM in a loop — it plans, takes actions (tool calls), observes results, then plans again. It keeps iterating until the task is done or it hits a stop condition.

Goal / Task
→
Plan next step
→
Call tool(s)
→
Observe result
→
Done?
# Agent pseudo-loop
while not done:
    response = llm.chat(history + tools)
    if response.tool_calls:
        results = execute_tools(response.tool_calls)   # run bash, read files, call APIs
        history.append(tool_results)
    else:
        return response.content                       # final answer
Multi-Agent Patterns
PatternDescriptionExample
Orchestrator + WorkersOne coordinator delegates to specialist sub-agentsPM agent → Dev agent + QA agent
PipelineAgents process sequentially, passing output forwardScraper → Summarizer → Report Writer
ParallelMultiple agents run simultaneously, results merged5 agents test 5 endpoints in parallel
Critic / ReviewerOne agent generates, another reviews/refinesCoder agent + Reviewer agent
Orchestration Frameworks
  • LangChain / LangGraph — flexible, graph-based agent flows
  • CrewAI — role-based multi-agent teams
  • AutoGen (MS) — conversational multi-agent
  • Claude Code — Anthropic's agentic coding agent
  • Pydantic AI — type-safe Python agents
Key Challenges
  • Hallucinated tool calls — model invents non-existent args
  • Infinite loops — always set max_iterations
  • Context overflow — long agent runs exhaust the window
  • Cost — many tool calls = many LLM calls = $$$
  • Observability — trace every step (LangSmith, Langfuse)
Real-world QA Automation Example

An agentic QA system that auto-triages CI failures:

Goal: "Investigate why the payment flow tests failed in CI run #8821"

Step 1: call get_ci_run(run_id="8821")
        → returns failed test names + stack traces

Step 2: call search_kibana_logs(trace_id="txn-abc123", time="last 1h")
        → returns Kafka event flow logs

Step 3: call search_codebase(class="LightSignDocumentsFlowCommand")
        → returns relevant Java source

Step 4: Reason: "transactionId=null → NullPointerException at line 47"

Output: "Root cause: transactionId not propagated from Kafka event.
         Fix: add null check + log warning in handleEvent().
         Linked issue: JIRA-4421"
